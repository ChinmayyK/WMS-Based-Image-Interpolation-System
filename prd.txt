PRD v1.0 identified problems, wrong assumptions, and technical risks. PRD v2.0 resolves every one of
them with concrete engineering decisions. This is not an incremental update — it is a complete

0.1 The Three Core Principles of This System
Principle 1 — Honesty Over Impressiveness
Every output the system produces must clearly communicate what it is: observed, interpolated-high-
confidence, interpolated-low-confidence, or rejected. A system that is honest about uncertainty is more
valuable than one that always generates an output. This principle is enforced at the architecture level,
not as a UI afterthought.
Principle 2 — Fail Loudly, Never Silently
Every failure mode — from a network timeout to a rejected interpolation frame — must produce a
logged, classified, user-visible signal. Silent failures are prohibited. The system must always know its
own state.
Principle 3 — Domain-First Engineering
Satellite imagery is not natural video. Every architectural decision — model selection, preprocessing
steps, validation thresholds, confidence scoring — must be derived from satellite imagery
characteristics, not from assumptions inherited from natural video pipelines.

1. Executive Summary
This document specifies a production-grade, accuracy-aware AI system that enhances the temporal
continuity of geostationary satellite imagery for WebGIS visualization. The system is designed to be
deployable, testable, and academically defensible — not a prototype or proof-of-concept.
Geostationary satellites (GOES-16/18, Himawari-8/9, INSAT-3D/3DR/3DS) observe the Earth at
intervals of 5–30 minutes, producing temporally sparse animation when visualised dynamically. The
system bridges this gap by generating statistically plausible intermediate frames using AI-based
interpolation, then classifying each generated frame by its reliability before delivering it to users.
ℹ NOTE: What This System Is and Is Not
IS: A visualization enhancement layer that makes satellite animation smoother and easier to
interpret.
IS: An accuracy-aware system that discloses the reliability of every frame it generates.
IS: A production-grade pipeline that fails safely, caches intelligently, and validates rigorously.
IS NOT: A scientific data product. Interpolated frames cannot replace real observations.
IS NOT: A weather forecasting system. No predictions of future atmospheric state are made.
IS NOT: A real-time system. Pipeline latency is 15–45 minutes for current architecture.

3.2 Hard Out-of-Scope Boundaries
✗ CRITICAL: Absolute Scope Exclusions
Scientific replacement of observed satellite data — interpolated frames are never to be treated as
real observations.
Numerical weather prediction or forecast generation of any future atmospheric state.
Quantitative derived-product interpolation (precipitation estimation, wind speed, SST).
Assimilation of interpolated frames into NWP models or any downstream scientific workflow.
Real-time operational use for weather warnings, aviation, or emergency response.
Polar-orbiting satellite data (Landsat, Sentinel-2) — revisit cycles are incompatible with frame
interpolation.
3D volumetric atmospheric modelling.
Any claim that interpolated frames represent ground truth or observed reality.

Rule OI-02 — Disclaimer Is Non-Dismissible
Every interpolated frame displayed in the WebGIS interface must carry a persistent watermark: "AI-
GENERATED — NOT OBSERVED DATA." This watermark must be encoded both in the video file
metadata and rendered as a UI overlay. The user must not be able to disable this watermark. The
watermark must be visible in screenshots and video exports.
Rule OI-03 — No Extrapolation Permitted
The system interpolates only between two existing observed frames. It must never generate a frame
that falls outside the temporal range of the two bounding observed frames. Extrapolation (predicting
beyond the last known frame) is architecturally prohibited.

Rule OI-04 — Gap Size Hard Limit
No interpolation may be attempted across a temporal gap exceeding 30 minutes. If two consecutive
observed frames are separated by more than 30 minutes, the pair is classified as a GAP and no
intermediate frames are generated. This limit is not configurable by the end user.

Rule DQ-02 — Sequence Segmentation on Gaps
When a gap (missing timestamp or gap > 30 min) is detected in the retrieved sequence, the sequence
must be split into two independent sub-sequences at that point. No interpolation may bridge sub-
sequence boundaries.
Rule DQ-03 — Radiometric Consistency Enforcement
If consecutive original frames show a radiometric jump exceeding 3 standard deviations of the session
mean inter-frame MAD, a CALIBRATION_SHIFT flag is raised. Interpolation depth for that pair is
capped at 1 frame and the pair is marked INTERPOLATED_LOW regardless of other metrics.

4.3 Model Governance Rules
Rule MG-01 — Model Qualification Gate (Phase 0 Enforcement)
No interpolation model may be used in the production pipeline until it passes the qualification gate on
the satellite benchmark dataset. The gate criteria are:
• PSNR ≥ 28 dB (mean across 50+ held-out frame pairs)
• SSIM ≥ 0.80 (mean across same set)
• LPIPS ≤ 0.15 (mean across same set)
• No catastrophic failure (SSIM < 0.40) on any single pair in the set
If RIFE 4.6 fails this gate, the fallback is Lucas-Kanade optical flow warping with bilateral filter post-
processing. This decision is documented and logged.
Rule MG-02 — Model Version Pinning
The interpolation model version must be pinned in the system configuration and recorded in every job's
audit log. Model updates must re-run the Phase 0 benchmark before deployment.
Rule MG-03 — Conservative Confidence on Domain Edge Cases
If the system detects a satellite-specific artefact (limb distortion zone, day-night terminator crossing,
calibration shift flag), the output confidence is capped at INTERPOLATED_MEDIUM regardless of
computed SSIM and flow metrics. This rule cannot be overridden at runtime.
4.4 API & System Governance Rules
Rule SG-01 — All API Inputs Validated
Every REST API endpoint must validate all input parameters against a defined schema before
processing. Invalid inputs return HTTP 422 with a structured error body — never HTTP 500.
Rule SG-02 — All Pipeline Errors Are Classified
Every exception in the pipeline must be caught, classified (RETRIABLE, NON_RETRIABLE,
PARTIAL_FAILURE), and logged with a structured log entry. Unhandled exceptions at any stage are a
testing failure.
Rule SG-03 — Job Audit Log Is Mandatory
Every pipeline job must produce a JSON audit log recording: job_id, input parameters, each stage's
start/end time and status, frame-level confidence assignments, and final output file paths. This log is
retained for 30 days.

Rule SG-04 — No Silent Data Substitution
If a frame is rejected or missing, the system must insert a labelled GAP placeholder. It must never
silently duplicate an adjacent frame to fill the gap without labelling it as such.

5.2 End-to-End Data Flow — Authoritative Version
Stage 1 — Job Submission
• User submits job via REST API: {satellite, layer, bbox, start_time, end_time, interpolation_depth,
region_preset}
• API validates inputs (Rule SG-01); rejects with 422 on schema violation
• Job record created with job_id, QUEUED status; audit log initialised
Stage 2 — Data Ingestion (Module 6)
• Check local cache; return cached frames if TTL valid
• Build WMS request list for all timestamps in range at native cadence
• Execute requests through rate-limit queue with exponential backoff
• Validate each downloaded frame (Rule DQ-01); re-request on failure (max 3 retries)
• Store to local cache with timestamp, bbox, source, checksum
Stage 3 — Preprocessing (Module 7)
• Spatial consistency check across all frames; reproject or reject misaligned
• Radiometric normalisation to 0–1 float range per session
• Gap detection; sequence segmentation at gaps > 30 min or > 3x nominal interval (Rule DQ-02)
• Fill-pixel masking; limb zone masking; terminator detection
Stage 4 — Interpolation & Validation (Modules 8 + 9)
• For each valid frame pair in each sub-sequence:
◦ → Check gap duration; reject pair if > 30 min (Rule OI-04)
◦ → Run input validation gate (Rule DQ-01)
◦ → Compute adaptive thresholds from session statistics
◦ → Apply Interpolation Engine (recursive bisection to target depth)
◦ → For each generated frame: compute flow consistency, SSIM, MAD
◦ → Apply Rule MG-03 domain overrides
◦ → Assign confidence label (Rule OI-01)
◦ → Reject frames below threshold; insert GAP placeholder (Rule SG-04)
Stage 5 — Video Generation (Module 10)
• Assemble full frame sequence: [OBSERVED, INTERP*, OBSERVED, GAP?, OBSERVED, ...]
• Generate dual output: original-only video and full interpolated video

• Generate HLS segments for streaming (optional, production mode)

• Frontend fetches metadata to drive confidence overlay and timeline

range first, then to [0,1].
• Band 2 (0.64 µm) reflectance: valid range 0–1.3; clamp to [0,1] before normalisation.
• Scan lines at image edges may contain fill values (-999 or 65535 in DN); mask before use.

• INSAT-3D TIR1 (10.8 µm): similar characteristics to GOES Band 13.

8.2 RIFE Fine-Tuning Strategy
8.2.1 Training Data
• Source: NOAA GOES-16 ABI Level-1b data from AWS S3 (s3://noaa-goes16/ABI-L1b-RadF/)
• Period: 1 calendar month (e.g., June 2023 — mix of convective and stratiform conditions)
• Band: Band 13 (thermal IR) — day/night valid, cloud motion clearly visible
• Volume: ~4,320 full-disk images (10-min cadence × 30 days). Download ~100 GB compressed.
• Split: 80% train / 10% validation / 10% test. Split by day — no temporal leakage.
8.2.2 Training Setup
• Input: Frame pairs (t, t+10 min). Supervised target: t+5 min (midpoint, withheld).
Page 21 of 53

• Loss: L1 reconstruction loss + perceptual loss (VGG-16 features) + flow smoothness
regularisation.
• Hardware minimum: GPU with 8 GB VRAM. Training time: ~12–24 hours.
• Fine-tuning strategy: Freeze RIFE backbone; train only optical flow and synthesis modules.
Reduces training time to ~4–6 hours.
8.2.3 Fallback — Lucas-Kanade Optical Flow Warping
• No training required. Deterministic. Always produces an output.
• Implementation: Compute dense optical flow (cv2.calcOpticalFlowFarneback) between frame
pair. Warp each frame toward the midpoint. Blend using alpha=0.5.
• Post-process with bilateral filter to reduce warping artifacts.
• Expected PSNR: 24–27 dB (lower than RIFE but physically more interpretable).
• This fallback always serves as a baseline comparison in the evaluation report.

System
ℹ NOTE: Terminology Correction from v1.0
This module was called "Accuracy Control" in the original proposal and "Validation" in PRD v1.0.
Its correct name is "Statistical Plausibility Module" — it verifies visual and statistical consistency,
NOT physical correctness.
Physical correctness cannot be verified without NWP comparison, which is out of scope.
All documentation, code, and UI labels must use this precise terminology.
9.1 Adaptive Threshold Calibration
Static thresholds fail for different regions, seasons, and cloud regimes. This module uses adaptive
calibration:
• At session start, compute baseline statistics from the first 10 original frames: mean inter-frame
MAD, mean inter-frame SSIM, optical flow magnitude distribution.
• Set session-specific thresholds as percentiles of these baseline distributions (e.g., MAD
threshold = 90th percentile of baseline inter-frame MAD).
• If < 5 original frames available (short time range), fall back to hardcoded global defaults.
• Log all threshold values used in the session audit log.

18. Operational Policies & Governance
18.1 Scientific Integrity Policies
POL-SCI-01 — Interpolated Data Is Never Scientific Ground Truth
No interpolated frame produced by this system may be cited, presented, or used as a scientific
observation. All interpolated frames are labelled synthetic in all outputs, all metadata, all API
responses, all UI displays, and all video files. Violation of this policy constitutes academic misconduct.
POL-SCI-02 — Limitations Must Be Disclosed in All Demonstrations
Every public demonstration, presentation, or viva defence of this system must explicitly state:
"Interpolated frames are AI-generated and statistically plausible — they are not observed satellite data
and do not represent what the atmosphere actually looked like at the interpolated time."
POL-SCI-03 — No Extrapolation
The system must never generate a frame temporally outside the bounding pair of observed frames.
Rule OI-03 is enforced architecturally.
18.2 Data Governance Policies
POL-DATA-01 — NASA GIBS Terms of Use
NASA GIBS data is freely available for use, including in applications, subject to NASA's Open Data
Policy. The system must credit GIBS in all outputs: "Satellite imagery courtesy NASA GOES-East via
NASA GIBS."
POL-DATA-02 — MOSDAC / ISRO Terms
INSAT data accessed via MOSDAC requires registration and is subject to ISRO's data policy. Verify
terms at mosdac.gov.in before any public deployment. For academic use within the college, this is
generally permitted with attribution.
POL-DATA-03 — Cache Retention
Downloaded satellite data frames are cached with a 24-hour TTL by default. Cached data must not be
distributed or used beyond the stated pipeline purpose. Cache must be purgeable via API endpoint
DELETE /api/v1/cache.
18.3 Code & Development Policies
POL-DEV-01 — Version Control
All code is maintained in a Git repository. No direct commits to main branch. All changes via pull
request. At least one team member reviews each PR before merge.

POL-DEV-02 — Dependency Management
All Python dependencies pinned in requirements.txt with exact versions. Node.js dependencies pinned
in package-lock.json. No floating version specifiers (no ">=", only "==") in production dependencies.
POL-DEV-03 — Model Checkpoint Management
Model checkpoints are stored outside the Git repository (use Git LFS or a separate S3 bucket).
Checkpoint SHA-256 hash is recorded in requirements.txt or a model_registry.json file. Any checkpoint
change requires re-running the Phase 0 benchmark.

Indian Ocean coverage.

Metric checking whether the total pixel intensity (proxy for cloud mass) is

variant).

PSNR/SSIM.

OGC Open Geospatial Consortium — the standards body for WMS, WMTS, WFS

Per-session scaling of pixel values to [0.0, 1.0] float32 range based on the

Recursive Bisection Interpolation strategy: generate the midpoint frame first, then midpoints of

interpolation model. Must pass Phase 0 benchmark before use.

Module 4 of this system. Assesses the visual and statistical plausibility of
Module

tOF Temporal Optical Flow — metric measuring the consistency of motion across
consecutive output frames relative to the original sequence.
TTL Time-To-Live — the duration for which a cached item is considered valid

WMS Web Map Service — OGC standard HTTP protocol for serving georeferenced
map images.
